# REALPROOF サブエージェント一式 — 導入README

## 何これ
君の「計画だけ投げて往復を減らす」を実現するためのサブエージェント定義。
役割を4体に分け、実装エージェントとレビューエージェントを**別インスタンス**にして相互チェックさせる。人間ゲート(SQL実行・push・デプロイ)は残す設計。

## ファイル配置
リポジトリ直下(`~/dev/forte-mvp`)にこの構造で置く:
```
.claude/
  rp-reference.md          ← 全エージェント共通の情報源(DB実カラム・絶対ルール)
  agents/
    rp-investigator.md     ← 調査(読み取り専用)
    rp-implementer.md      ← 実装
    rp-reviewer.md         ← レビュー(別人格・容赦なく指摘)
    rp-db-guard.md         ← DB安全番(SQL提示のみ・実行しない)
```
置いたらClaude Codeを再起動。`rp-reference.md` の `<PROJECT_REF>` はそのまま(実値はファイルに書かない方針)。

## 4体の役割分担
| エージェント | ツール | 何をする |
|---|---|---|
| rp-investigator | Read/Grep/Glob | Phase 0の現物確認・症状の一次調査。推測せず事実を報告 |
| rp-implementer | Read/Write/Edit/Bash | 1フェーズ分を最小変更で実装+`tsc`まで |
| rp-reviewer | Read/Grep/Bash | 実装をCLAUDE.md規約で機械チェック。甘くしない |
| rp-db-guard | Read/Grep | SQLを実行せず「確認→本番→検証」の安全SQLに整える |

## 基本の回し方(メインセッション=君が指示するリード)
1. 「まず rp-investigator で Phase 0 を調べて」→ 事実が返る → 🛑
2. 「rp-implementer で Phase 1 を実装」→ 変更+tsc → 🛑
3. 「rp-reviewer にレビューさせて」→ PASS/要修正 → 要修正なら実装に差し戻し
4. DB変更があれば「rp-db-guard に安全SQLを出させて」→ **君がSupabase SQL Editorで手動実行**
5. PASS後、**君が**コミット/GitHub Desktopでpush/デプロイ

= コードの往復は自動、危険な3操作(SQL/push/デプロイ)だけ手動。

## 呼び出し方(Claude Code内)
- 自動委任: 「Phase 0を調査して」と言えばdescriptionに従い自動でrp-investigatorが動く
- 明示指定: `@rp-reviewer で今の差分を見て` のように名前指定すると確実にそのエージェントが動く

## 次段階: Agent Teams化(君の「最終的にまとめたい」への回答)

### できる。ただし今のルールと1点衝突する
Agent Teams = リード1体+チームメイト複数が**共有タスクリストで並列協調**する上位機能。サブエージェント(一方向の委任)と違い、チームメイト同士がメッセージし合う。

**衝突点**: チームメイトが**並列でファイルを書く**場合、公式は**worktree(作業ディレクトリ分離)を強く推奨**。でも君のルールは「ブランチ・worktree禁止・main直接コミット」。並列書き込み×worktree禁止は、ファイル競合(レースコンディション)を招く。

### だから現実的な線引き
- **並列で"書く"のはチーム化しない**(君のmain直コミット規律を守る)。実装は基本1本、サブエージェントで役割分担するだけ
- **並列で"読む/レビューする/調べる"はチーム化が効く**: 例=「セキュリティ監査・パフォーマンス・テスト網羅」を同時に走らせる、複数仮説のバグ調査を並列で潰す。ここはworktree不要でハマる
- チームは**トークン消費がチーム人数に比例**して増える。ルーチンには単一セッションが安い

### 有効化(実験的機能)
`settings.json` の env に `"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"` を追加して再起動。
(実験的につき、セッション再開・タスク調整・終了周りに既知の不安定さあり)

### 提案する到達点
- **実装フロー**: サブエージェント(調査→実装→レビュー→DB安全番)のまま。ここはチーム化しない
- **監査/大規模調査**: Agent Teamsで並列(例: リリース前に「規約監査・型/ビルド・DB整合・UXフロー」を各teammateが同時点検)
- worktree禁止ルールを一部緩めてもいいと判断したら、並列実装のチーム化も解禁できる。そこは別途相談

## 重要な留保
サブエージェントもチームも「動くか/規約通りか」は見るが、**プロダクト判断(ゲートにしない・核動線を食わない等)は見ない**。仕様の正しさは今まで通り君と設計で詰める。エージェントは判断を増幅するだけで、生まない。
